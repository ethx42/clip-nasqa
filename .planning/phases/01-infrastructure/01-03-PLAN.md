**Expected outcome:** SST deploys successfully and prints outputs including `apiUrl` and `tableName`.

**If lint fails:** Fix ESLint errors in packages/frontend and re-run.

**If typecheck fails:** Fix TypeScript errors in packages/core or packages/functions and re-run.

**If sst deploy fails with SST Ion API errors** (these are the open questions from research):
- `dataSource: "NONE"` not accepted → Add `const noneDS = api.addDataSource({ name: "noneDS" })` with no lambda/dynamodb property and use `noneDS.name` as the dataSource for the subscription resolver.
- `api.apiKey` not a valid property → Remove from return statement; use `api.url` only.
- `link: [table]` not supported inside addDataSource lambda → Remove `link` and rely on `TABLE_NAME` env var; add IAM permissions manually via `transform` if needed.
- `"Query _empty"` resolver name format wrong → Try `"Query._empty"` with a dot separator.
- `code:` path to resolver file → If SST doesn't accept the file path string, inline the resolver code as a template literal string in sst.config.ts.

After successful deploy, capture the outputs:
```bash
cat .sst/outputs.json
```

Log the `apiUrl` and `tableName` values — these will be needed in Phase 2 and 3.
