Create the `infra/resolvers/` directory and write two AppSync APPSYNC_JS runtime resolver files.

**infra/resolvers/subscription-onSessionUpdate.js** — Enhanced filter resolver for Subscription.onSessionUpdate:
```javascript
import { util, extensions } from '@aws-appsync/utils';

export function request(ctx) {
  return { payload: null };
}

export function response(ctx) {
  if (!ctx.args.sessionSlug) {
    util.error("sessionSlug is required", "Unauthorized");
  }
  extensions.setSubscriptionFilter(
    util.transform.toSubscriptionFilter({
      sessionSlug: { eq: ctx.args.sessionSlug }
    })
  );
  return null;
}
```

Critical: This resolver uses the NONE data source (no backing resource). It MUST use `extensions.setSubscriptionFilter` — NOT argument-based filtering. Once setSubscriptionFilter is called, argument-based filtering is disabled and cannot be bypassed by a null argument.

**infra/resolvers/query-stub.js** — Stub resolver for Query._empty so AppSync has at least one resolvable query field:

```javascript
  return { operation: 'GetItem', key: { PK: { S: '_stub' }, SK: { S: '_stub' } } };
}

export function response(ctx) {
  return null;
}
```

Note: query-stub.js will be wired to a DynamoDB data source in sst.config.ts. It exists only for schema health validation — it is never called in production.

**sst.config.ts** — Replace the empty run() body with the full resource definition. Keep the existing app() config block unchanged.

```typescript
/// <reference path="./.sst/platform/config.d.ts" />

export default $config({
  app(input) {
    return {
      name: "nasqa-live",
      removal: input?.stage === "production" ? "retain" : "remove",
      protect: ["production"].includes(input?.stage),
      home: "aws",
    };
  },
  async run() {
    // 1. DynamoDB single-table — PK/SK composite key, TTL attribute, on-demand billing
    const table = new sst.aws.Dynamo("NasqaTable", {
      fields: {
        PK: "string",
        SK: "string",
      },
      primaryIndex: { hashKey: "PK", rangeKey: "SK" },
      ttl: "TTL",
      transform: {
        table: {
          billingMode: "PAY_PER_REQUEST",
        },
      },
    });

    // 2. AppSync GraphQL API — schema-first, API key authentication
    const api = new sst.aws.AppSync("NasqaApi", {
      schema: "infra/schema.graphql",
    });

    // 3. DynamoDB data source (for stub Query resolver)
    const dynamoDS = api.addDataSource({
      name: "dynamoDS",
      dynamodb: table,
    });

    // 4. Lambda data source (for mutations — Phase 3 adds real handlers)
    const lambdaDS = api.addDataSource({
      name: "lambdaDS",
      lambda: {
        handler: "packages/functions/src/resolvers/index.handler",
        environment: {
          TABLE_NAME: table.name,
          AWS_NODEJS_CONNECTION_REUSE_ENABLED: "1",
        },
        link: [table],
      },
    });

    // 5. Stub resolver for Query._empty (DynamoDB data source — always returns null)
    api.addResolver("Query _empty", {
      dataSource: dynamoDS.name,
      code: `infra/resolvers/query-stub.js`,
    });

    // 6. Subscription resolver — enhanced server-side filter (NONE data source)
    // CRITICAL: Uses extensions.setSubscriptionFilter for session isolation.
    // Argument-based filtering alone has a null-slug bypass vulnerability.
    api.addResolver("Subscription onSessionUpdate", {
      dataSource: "NONE",
      code: `infra/resolvers/subscription-onSessionUpdate.js`,
    });

    return {
      apiUrl: api.url,
      tableName: table.name,
    };
  },
});
```

**Important implementation notes:**
- If `dataSource: "NONE"` is not accepted by SST Ion's addResolver, try: first call `api.addDataSource({ name: "noneDS" })` with no lambda/dynamodb property (a bare NONE data source), then reference `noneDS.name` as the dataSource.
- If `api.addResolver` expects a different field format (e.g., `"Subscription.onSessionUpdate"` with a dot instead of space), adjust accordingly — check SST Ion docs at runtime.
- If `dynamoDS.name` is not a string property, use the string `"dynamoDS"` directly (SST typically exposes the name string).
- If `link: [table]` inside addDataSource lambda is not supported, set `TABLE_NAME: table.name` as env var (already present) and the Lambda will use the env var. IAM grants may need to be added manually via `transform` if link doesn't work inline.
- The `code: \`infra/resolvers/...\`` path is relative to where sst.config.ts lives (the repo root). This is the correct path convention for SST Ion.
