   - `downvoteQuestionAction(sessionSlug: string, questionId: string, fingerprint: string, remove?: boolean)`: Same pattern with DOWNVOTE_QUESTION.
   - `restoreQuestionAction(sessionSlug: string, hostSecretHash: string, questionId: string)`: Same pattern with RESTORE_QUESTION.
   All actions should be `'use server'` and handle errors (catch and return error string or throw).

   - Update the existing `QUESTION_UPDATED` case to handle `isBanned` and `isHidden` fields on the question — when a question update arrives with `isBanned: true` or `isHidden: true`, update the question in state accordingly.
   - Add `PARTICIPANT_BANNED` case: add `payload.fingerprint` to `bannedFingerprints` Set. Return new state with updated Set.
   - Update the existing upvote handling: make upvote toggleable (retractable). The current upvoteQuestion mutation should handle `remove: true` — check if this is already supported. If the existing `QUESTION_UPDATED` handler already updates `upvoteCount`, this may already work. The key change is client-side: track downvoted question IDs alongside voted IDs.
   - Ensure `QUESTION_UPDATED` payload now includes `downvoteCount`, `isHidden`, `isBanned` fields and the reducer updates questions with these values.
