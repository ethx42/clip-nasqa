```typescript
interface UseSessionMutationsParams {

  onReplySuccess?: () => void;
}
```

Returns:

```typescript
interface SessionMutations {
  handleUpvote: (questionId: string, remove: boolean) => Promise<void>;
  handleDownvote: (questionId: string, remove: boolean) => Promise<void>;
  handleAddQuestion: (text: string) => Promise<void>;
  handleReply: (questionId: string, text: string) => Promise<void>;
}
```

STALE CLOSURE FIX (critical requirement):
The existing downvote handler has a stale closure bug: `state.questions.find(...)` inside the rollback IIFE captures the questions array from the render when the handler was created. Under concurrent votes, this reads stale data.

Fix: Accept a `questionsRef` (React ref pointing to current questions array). The page orchestrator keeps this ref synced via `useRef` + `useEffect`. Inside rollback handlers, read `questionsRef.current.find(...)` instead of `state.questions.find(...)`. This reads live state without capturing a snapshot.

Implementation for each handler — move the exact logic from session-live-page.tsx:

- `handleUpvote`: Optimistic dispatch + vote tracking + safeAction + rollback on failure. Keep VOTE_CONFLICT silent handling.
- `handleDownvote`: Optimistic dispatch + downvote tracking + mutual exclusion with upvote. Use `questionsRef.current` for reading current downvoteCount in rollback.
- `handleAddQuestion`: Optimistic question with temp `_opt_` ID + safeAction + rollback on failure. Call `onSubmitSuccess?.()` after successful action.
- `handleReply`: Optimistic reply + safeAction + rollback. Use `isHostReply` param (false for participant, true for host). Call `onReplySuccess?.()` after successful action.

Error handling is internal — each handler calls `toast.error(result.error, { duration: 5000 })` on failure. Use `useTranslations("actionErrors")` inside the hook for the networkError fallback string.

Wrap each returned handler in `useCallback` with appropriate dependencies to avoid unnecessary re-renders.
